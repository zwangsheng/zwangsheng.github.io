java 程序的执行过程

.java -(javac)> .class -(JVM)> 方法区
java 文件被javac翻译成class文件，由jvm加载到方法区，可以被JVM的执行引擎进行执行，执行引擎执行的时候，会翻译成操作系统的机器指令

jvm的本质

jvm是虚拟化的类操作系统
操作系统---JVM
CPU ---执行引擎
内存 ---堆，栈
缓存 ---操作数栈

java的语言特性

跨平台 --- 有适用于不同操作系统的JVM
跨语言 --- jvm执行的并不是java语言，而是class文件，class文件具有很好的向后兼容性

jvm的内存模型


---线程私有---

java虚拟机栈

先进后出的数据结构
JVM运行的时候存放当前线程执行的方法所需的数据，指令，返回地址
生命周期随着线程
–Xss 调整大小

栈帧

方法调用的单位，方法的进入和退出，对应着栈帧的入栈和出栈

局部变量表

存放局部变量，以槽为单位，一个槽对应32位，对于64位的数据采用高低位占用的方式，对于引用类型的数据，一般存放引用（开启逃逸分析以后，会直接在栈上分配）
存在jvm的优化策略，上个方法的操作数栈和下个方法的局部变量表共享一块空间

变量槽复用

当一个局部变量出了它的作用域，它的变量槽可以被其他变量复用，算是JVM的优化操作

操作数栈

存放java方法需要的操作数，方法刚开始的时候为空

动态连接

java语言的多态性

返回地址

正常返回 程序计数器当中的地址
异常返回 异常表

本地方法栈

类似java虚拟机栈，不写，无非运行本地native方法

程序计数器

较小的内存空间，当前线程执行的字节码的行数（为啥要，CPU时间片轮转，挂起的时候，回来还能知道执行到哪了）
分支，异常处理，循环，跳转依赖于程序计数器
唯一个不会OOM的区域
---线程共有---

堆

几乎所有对象都在这里分配（1.栈上分配2.TLAB）

方法区

存储每一个类的结构信息（元信息）
jdk8 移除，在堆外内存放了一个元信息空间来取代方法区

常量池

class 常量池（静态常量池）

运行时常量池

字符串常量池

String 的使用
String str1 = "abc";
String str2 = new String("abc");
String str3 = new String("abc").intern();

经典案例了
str1 引用字符串常量池，没有塞一个再用
str2 堆上分配了一个对象，对象引用常量池
str3 引用字符串常量池，没有塞一个再用
str1 ！= str2
str1 == str3

JVM启动，根据参数进行申请内存，进行运行时数据区的初始化，加载类到方法区，执行方法

对象的内存布局

对象的访问方式

句柄

在堆内维护一个句柄池，栈上的引用指向句柄池，句柄池有实例的地址和类的地址
好处：移动对象位置后，不用修改引用指向，只需要修改句柄池
坏处：多了一步访问

直接指针（hotspot）

栈上引用直接指向对象实例，对象实例的对象头维护一个指向类类型的指针
好处：直接了当的访问
坏处：对象的修改，对引用影响大

判断对象存活

引用计数法（python）

多一条引用+1，少一条引用-1
存在循环引用，需要额外的方法解决

可达性分析

从GC Roots出发，一路经过的对象认为存活，没经过认为不可存活

GC Roots
  1. 局部变量表的引用
  2. 方法区的静态引用
  3. 方法区的常量引用
  4. native栈引用
  5. 对synchronized持有的对象
  6. 跨代引用
class的回收条件
  1. 无该类实例
  2. 该类的加载器被回收
  3. class没有被反射调用
  4. 参数开启

引用类型

强引用

=，除非断开不然不回收

软引用 SoftReference

先回收其他对象，如果还不够就回收软引用

弱引用 WeakReference

发生垃圾回收就会被回收

虚引用 PhantomReference

随时被回收

逃逸分析

参数控制，默认开启
如果一个对象逃不出这个方法，那么会采取栈上分配

大对象直接进入老年代

  1. 避免大量的内存复制
  2. 避免提前垃圾回收

长期存活进入老年代

默认15岁，CMS 6，存活一次加一岁

动态年龄分配

Survivor相同年龄的对象到达空间的一半，那么大于等于这个年纪的对象进入老年代

空间分配担保

发生minor GC之前，检查老年代的剩余空间是否大于新生代所有对象的空间
检查老年代最大的大小，满足新生代平均大小，可以就minorGC，不然直接fullGC

垃圾回收算法

复制算法（Copying） - 新生代

内存分成两块，每次使用一块，垃圾回收时，活的移到另外一块，直接清空之前那块
实现简单，运行高效
空间浪费

apple式回收

分成 8:1:1 的eden from to
回收eden 和上次存放存活对象的那块，存活的对象移到另外一块上
浪费少，高效

标记-清除（Mark-Sweep）CMS

扫描标记不可活对象
扫描回收不可活对象
扫描了两边，回收效率低，会造成大量不连续的空间碎片

标记-整理（Mark-Compact）

扫描比较不可活对象
扫描移动存活对象到内存一侧
清空另一侧
效率低，直接指针访问方法需要修改大量的指针

垃圾回收器

Serial/Serial Old

单线程的垃圾收集器，新生代和老年代版本
不存在线程切换，当空间小的时候效率高

Parallel Scavenge/Parallel Scavenge

多线程，关注吞吐量的收集器
吞吐量（用户时间/（用户时间+垃圾回收时间））

ParNew

多线程，和CMS搭配工作

Concurrent Mark Sweep （CMS）

最短停顿时间，标记清除

初始标记

stop the word
标记GC Roots直接可达的对象
很快

并发标记

从GC Root关联的对象开始，遍历全部

重新标记

标记并发那段时间的改动
很快

并发清除

清除不可达对象

CPU敏感

CPU < 4 停顿感知很明显

浮动垃圾

并发清理的过程中会产生新的垃圾，意味着CMS需要预留空间 92%

空间碎片

由于标记-清除算法，会产生碎片垃圾，有个参数，允许分配不了大对象的时候进行内存碎片整理

Garbage First(G1)

利用region的思想取代分代思想，实现可预测停顿

Region

region具有种类，可能是Eden，Survivor，Old，Humongous（大对象，region的容量一半，一般作为老年代对待）

Top at Mark Start（TAMS）

每个Region中有两个TAMS，用来记录并发回收过程产生的新对象，认为他们不需要被回收

Rset 记忆集 解决跨代引用

记录跨代的引用
会占据部分堆空间

cardTabel 卡表 解决Rset寻找问题

数组记录那块区域之间存在跨代引用

初始标记

标记GC Roots 直接可达的对象，修改TAMS的值
可以借用minorGC的时候完成

并发标记

GC Root开始进行可达性分析，递归扫描整个堆，找出要回收的对象

最终标记

处理遗漏的对象

筛选回收

更新region的回收列表，按照价值进行回收，达到可预测停顿的目标，将存活对象移到新的region，删掉旧的

空间整合

标记整理 -复制算法

追求停顿时间

三色标记

  1. 黑色 根对象或者该对象和其自对象都被扫描过的对象
  2. 灰色 本身被扫描过，自对象没有被扫描
  3. 白色 未被扫描的对象

三色标记漏标的处理办法

Incremental Update CMS

关注引用的增加
当一个白对象被黑对象引用，将黑标灰，重新扫描灰

snapshot-at-the-beginning（SATB）G1

关注引用的删除
拍一个快照，发现灰与白的引用消失以后，按照快找照进行对比标记

安全点

GC 线程通知业务线程要进行GC，而不是强制打断
业务线程一直轮训GC的标识位，发生GC之后，到达一些安全点之后，会把自己挂起
安全点一般是方法跳用，循环跳转，异常跳转，不会和对象引用有关

安全区

在这块代码片段内，引用不会发生变化
GC不会管在安全区里面的线程
线程出安全区之前要轮训查看是否完成GC了
