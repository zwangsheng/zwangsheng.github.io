MySql的处理层

  2. 解析查询语句，创建一个解析树

SQL语句解析


执行引擎

Innodb

事务存储型引擎，完全支持ACID特性，支持行级锁

MyISAM

非事务存储，适合只读类应用

Innodb和MyISAM的区别

CSV

所有列都不能为null，可以直接在文本上进行修改，不支持索引

锁

表锁

开销小，上锁快，不会出现死锁
锁粒度大，发生冲突的概率高，并发度低

行锁

开销大，上锁慢，会出现死锁
锁粒度小，发送冲突的概率小，并发度高

间隙锁

介于表锁和行锁之间

行锁表锁使用场景的讨论

表锁适合以查询为主，只有少量按照索引进行更新
行锁适合大量的按索引条件并发更新的不同数据，拥有并发查询的能力

Innodb的上锁

lock in share mode 读锁
for update 写锁
两个事务不能锁同一个索引
insert ，delete ， update 在事务中都会自动默认加上排它锁
行锁只有遇到索引才能成为行锁，不然升级成为表锁

事务

事务的特性

原子性

操作要么全部成功要么全部失败

一致性

操作前后的结果保持一致

隔离性

事务与事务之间保持隔离

持久性

事务一旦提交就被持久化

事务的并发问题

脏读

事务A读取事务B的数据，事务B进行回滚，导致A读到了脏数据

不可重复读

事务A多次读取相同的数据，事务B在这个过程中修改了数据，导致多次读取不一致

幻读

事务A多次读取大量数据，事务B往中间查了新的数据

事务的隔离级别

读未提交

读取未提交的事务的数据 --写允许读

读已提交

读取已提交的事务的数据 --写静止读

可重复读

事务开启之后，不允许写操作 --读静止写

串行化

所有事务串行化

业务设计

三大范式

  1. 所有字段值都不可再分
  2. 表中的每个数据都要与主键相关
  3. 都与主键之间相关，而不是间接相关

反范式设计

添加冗余来避免热点数据的连表查询

索引

聚集索引

索引项的排序方式和表中记录的排序方式一样

一张表只能有一个聚集索引

使用场合

返回的字段按照这个排序
查询结果返回一个区间的值
查询结果返回某值相同的大量结果集

非聚集索引

使用场合

唯一性比较高的时候
获得的数据量少的情况

索引失效的情况

  1. like的%在前面
  2. 使用索引列进行计算
  3. 对索引使用了函数
  4. 对索引使用了!=

存储结构

B-Tree

所有的键值分布在这个树的每个节点中

搜索可能在非叶子节点结束

所有的叶子节点在同一层，按照升序排列

B+Tree

树的非叶子节点不存放数据

叶子节点存储数据和关键词，非叶子节点的关键字也会沉到叶子节点进行排序

叶子节点形成双向环状链表

执行计划

帮助分析优化SQL语句

详解

ID列

表示查询select子句的顺序

id相同时，从上到下执行

id不同时，id大的先执行

Select_Type列

区别普通查询，联合查询，子查询

Table列

表示数据从哪张表来

Type列

访问类型，衡量sql好坏的指标
system > const > eq_ref > ref > fulltext > ref_or_null > index_merge >unique_subquery > index_subquery > range > index > ALL

System

表里只有一条数据

Const

通过索引一次就找到了

eq_ref

唯一性索引扫描

ref

非唯一性索引扫描

range

只检索给定范围的行，使用一个索引来选择行

index

查询结果全是索引列的时候，虽然查了全表，但是也只是索引

ALL

遍历全表进行查询

possible_keys

可能使用的索引

Key

实际使用的索引

Key_len

表示索引使用的字节数

Ref

显示索引的那一列被使用了

Rows

估算找到记录所需要读取的记录条数
