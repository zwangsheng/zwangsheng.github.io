进程 进程是程序运行资源分配的最小单位

线程 线程是CPU调度的最小单位，需要依赖进程存在

并发 交替运行，单位时间

并行 同时进行

守护线程 一般是支持性线程，做辅助工作，当不存在非守护线程的线程时，JVM自动结束

CPU时间片轮转机制

CPU将时间切片，给进程分配时间，如果时间片用尽，CPU将分配给其他进程，如果时间片没用尽就阻塞或结束，CPU也会进行转换
因为进程的上下文切换需要消耗资源，所以存在时间片时长的问题
如果时间片太小，那么每次做完都会进行上下文切换，CPU的时间很大一部分浪费在切换上
如果时间片太长，交互请求会变差

run()

业务逻辑实现的地方，调用后不会开启新的线程来处理，当作一个普通方法使用

start()

让一个线程进入队列等待CPU，分配到CPU以后会执行run()方法，不可重复执行

yield()

让当前线程让出CPU使用权，不会释放锁资源，让出后立刻进入就绪状态，可能被立刻选中

sleep()

线程进入休眠，不会释放锁资源

join()

把指定线程加入到当前线程，将交替运行的线程改为顺序进行
B线程调用了A线程的join，直到A执行完，B才能继续执行

wait()

线程进入WAITING状态，只有等待另外线程通知，被中断或者超时才会返回，会释放锁

notify()

通知在对象上等待的一个线程，让其从wait上返回，如果获得锁就运行，没获得就进入WAITING状态

notifyAll()

通知对象上等待的所有线程

抢占式调度 系统通知

协同式调度 线程本身控制，通知系统进行切换

线程共享

synchronized

对象锁

  1. 同步代码块，锁住对象实例
  2. 加在方法上

类锁

  1. 同步代码块，锁住类
  2. 加在静态方法上

锁膨胀

volatile

保证不同线程对修饰变量的可见性，但不能保证原子性

ThreadLocal

ThreadLocal为所有线程提供变量副本
每个线程内部维护了一个ThreadLocalMap，key 是 ThreadLocal的实例

ThreadLocal可能存在的内存泄漏问题


key 与 ThreadLocal实例是弱引用，一旦发生GC就会被回收
那么ThreadLocalMap就回存在key为null的value，无法被访问到，可能会造成内存泄漏
只有currentThread结束，才能被回收

如果ThreadLocal的实例与key为强引用，那么当栈上的ThreadLocalRef不再指向ThreadLocal之后，key与ThreadLocal还是强引用，依然造成内存泄漏
如果ThreadLocal的实例与key为弱引用，那在接下来的set，get，remove的过程中还有可能对key为null的value进行清除
所以根源并不是弱引用，而是ThreadLocalMap的生命周期和线程一样长，如果不及时删除对应的key就回导致内存泄漏

线程协作

等待通知机制

等待方

  1. 获得锁
  2. 条件不满足就wait，醒了以后还得检查条件
  3. 条件满足进入业务

通知方

  1. 获得锁
  2. 改变条件
  3. 通知所有等待的线程

线程并发的工具类

ForkJoin

RecursiveAction，用于没有返回结果的任务

RecursiveTask，用于有返回值的任务

工作密取

线程工作复杂度不同，有的线程先完成任务，那么从别的线程的队列头开始取工作完成

使用流程

CountDownLatch


线程等待其他线程完成工作之后才开始进行
通过一个计数器进行实现，每当一个线程完成任务就进行-1，到达0之后，使用了await()方法的线程就可以往下运行
只能用一次

CyclicBarrier


等待所有线程到达以后，进行放行
线程达到后调用await()告诉CyclicBarrier已到达，然后将阻塞
可以传入barrierAction，用来所有线程都到达之后先执行这块代码再放行
允许反复使用

Semaphore


一般拿来做流控
acquire() 获取许可证
release() 归还许可证

Exchange


用于线程之间的数据交换，提供一个同步点，在这个同步点，允许线程相互交换数据
一个线程调用exchange()，会等待第二个线程也执行exchange()

原子类Atomic

利用CAS，volatile进行保证原子性

ReentrantLock

显示加锁
可重入，公平非公平靠参数控制，响应式中断
state 保证可重入
fair 控制公平非公平
读写锁的实现通过16位偏移

读写锁的升降级

在持有写锁的过程中，获得读锁，然后释放写锁

LockSupport

定义了公共的静态方法，提供了基本的线程阻塞和唤醒功能

CLH

队列锁是一种基于链表的可扩展，高性能，公平的自旋锁
申请的线程仅仅只在本地变量上自旋，不断的轮询前驱节点的状态，前一个节点释放锁了，就结束轮询获得锁

CLH流程

  1. 创建一个Node，将标识位locked设为true，有一个myPred标识前驱节点
  2. 对CLH的tail调用GetAndSet方法，CAS的让自己成为队列的尾部，同时获得前驱引用
  3. 轮询前驱节点的locked变成了false，前驱节点被回收，自己成为前驱节点，同时获得锁

AQS

int 成员变量state作为同步状态，内置的FIFO队列来完成线程的排队，Node存放线程信息

CAS


获取期望值，在更新之前，判断期望值是不是符合，符合就进行操作

ABA问题

中间发生了变化，但是结果相同，操作不可知
解决办法，加上版本号

只能保证一个共享变量的原子操作

ConcurrentHashMap初始化只需要一个线程

采取多线程的扩容

生产者和消费者不直接通讯，通过阻塞队列进行通讯，阻塞队列相当于缓冲区，平衡了生产者消费者的能力

线程池

  1. 资源消耗降低
  2. 提高响应速度
  3. 提高线程的可管理性

corePoolSize 线程池核心线程数

maximumPoolSize 线程池最大线程数

keepAliveTime 线程空闲存活时间，对非核心线程才生效

workQueue 保存工作的阻塞队列

threadFactory 创建线程的工厂

RejectedExecutionHandler 拒绝策略，阻塞队列满了且达到最大线程数时执行的策略

线程池工作机制

  1. 当前线程少于核心数，就造核心线程
  2. 当前线程等于核心数，任务塞进阻塞队列
  3. 阻塞队列满了，开新的线程处理
  4. 线程数达到最大线程数，采取拒绝策略

execute() 无返回值的提交

submit()有返回值的提交

shutdown() 设置线程池状态SHUTDOWN，中断所有没在执行的线程

shutdownNow() 设置线程状态STOP，尝试关闭所有线程，返回还在做的列表

isShutdown() 并不能代表线程池真的完全关闭

isTerminaed() 线程池完全关闭

CPU密集型 分配 CPU + 1线程的线程池

IO密集型 分配CPU * 2线程的线程池

FixedThreadPool 固定线程数的线程池 有界队列

SingleThreadExecutor 单线程线程池 有界队列

CachedThreadPool 缓存线程池，无核心线程数，无边界队列，极端情况下会炸

WorkStealingPool 工作窃取，具体看forkJoin
